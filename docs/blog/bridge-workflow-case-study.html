Case Study Cross-Platform Real Estate March 2026

Why I built a 3-platform bridge instead of picking one

A real-world architecture decision: how connecting Zapier, n8n, and Make saved a real estate agency $150/month while adding AI lead qualification they couldn't have built any other way.

The situation

A real estate agency approached me with a clear problem: their agents were spending 3+ hours every day manually reviewing lead form submissions. High-value leads sat uncontacted for hours while agents waded through unqualified inquiries.

They already had Zapier running — 30+ Zaps connecting Facebook Lead Ads to their CRM, scheduling follow-up tasks, notifying the team. It was working. But they wanted AI lead qualification added to the pipeline without rebuilding everything.

The constraint: "We're not migrating our existing Zapier setup. It took 6 months to build and everything depends on it. We just want AI scoring added on top."

This constraint turned out to be a design opportunity, not a limitation. Instead of recommending a full platform migration, I designed a three-platform bridge architecture where each platform handles exactly what it does best.

The architecture decision

The core insight: you don't need to pick one automation platform. You need to pick the right platform for each segment of your pipeline.

Why each platform was chosen

Zapier: the existing foundation

Zapier stays for one reason: migration risk. The agency has 30+ Zaps built over 6 months. Each Zap has edge cases, tested filters, and team members who know how to use them. Rebuilding them in another platform isn't just technical work — it's organizational risk.

Zapier's 6,000+ app integrations and its team's existing familiarity make it the right anchor for the capture layer. The only addition: a "Webhooks by Zapier" action that POSTs qualifying leads to n8n. One new step. Zero migration.

n8n: the AI processing engine

n8n runs self-hosted on a $6/month DigitalOcean Droplet. This is the key cost decision:

OptionAI ProcessingCost at 1,000 leads/mo
Zapier AI ActionsBuilt-in (limited)~$100–200/mo
Make + OpenAI HTTPComplex setup~$40–60/mo
n8n self-hosted + OpenAINative LangChain nodes~$7/mo

n8n's AI Agent node, Code node, and native OpenAI integration make the AI qualification layer genuinely powerful. The GPT-4o-mini scoring prompt analyzes budget, timeline, property type, and message sentiment to produce a 1-10 score with reasoning and a recommended action — in under 3 seconds per lead.

Make: the BI reporting layer

Make's Router module is uniquely well-suited for the final distribution step. Hot leads simultaneously hit Google Sheets, Slack, and Google Calendar — all in parallel, in a single scenario. Make's operation pricing makes this far cheaper than equivalent Zapier multi-step actions at volume.

Beyond cost, the agency now gets:

  • Sub-3-second lead scoring — agents know a lead's priority before they even open their CRM
  • AI-drafted first email — personalized to each lead's specific property interest and budget
  • Hot lead Slack alert — agents receive a notification with full context and the recommended action
  • Zero migration risk — existing 30+ Zaps remain untouched and continue working
  • Independent failover — if any platform goes down, the others continue functioning independently

Key technical decisions

Webhook communication between platforms

The platforms communicate via HTTP webhooks — the universal lingua franca of automation. Zapier POSTs to n8n's webhook URL; n8n POSTs to Make's webhook URL. Each platform is independently deployed and testable. You can curl any webhook to test a segment in isolation.

Setup order matters

Always configure downstream-first: Make → n8n → Zapier. Each platform's webhook URL is only available after it's been activated, so you build in reverse order to have each URL ready before the upstream platform needs it.

Failover design

n8n uses a try/catch in its Code node to handle AI parse errors gracefully. If n8n is unreachable, Zapier's error handler writes the raw lead to a fallback Google Sheet. Make's auto-commit means partial runs save what they completed. No single point of failure kills the entire pipeline.

When to use this pattern

This bridge architecture is the right choice when:

  • A client has existing automations they can't migrate
  • You need AI/LLM processing that's cost-prohibitive in Zapier or Make
  • Different teams own different parts of the pipeline (marketing team → Zapier, dev team → n8n)
  • High-volume operations would be expensive in a per-task model but cheap in a per-execution model
  • You need custom JavaScript logic that no-code tools can't handle
The bridge pattern isn't about having three platforms for its own sake. It's about letting each platform do what it's genuinely best at — and connecting them at the cheapest possible seam (webhooks).
